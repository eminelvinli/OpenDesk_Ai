# 🚀 OpenDesk AI - Technology Stack Summary

This document outlines the complete technology stack used across the OpenDesk AI monorepo. Every technology was chosen to balance **cross-platform desktop performance, high-concurrency network routing, and advanced AI orchestration**.



## 1. Desktop Client (`/desktop_client`)
**Core Purpose:** Low-level OS interaction, ultra-fast screen capturing, and minimal resource consumption on the user's machine.
* **Core Language:** **Rust** (Chosen for memory safety, C-like performance, and native OS API access).
* **UI Framework:** **Tauri** (Creates incredibly small, cross-platform binaries using the OS's native webview, avoiding the heavy footprint of Electron).
* **Screen Capture:** `xcap` or `scrap` (Rust crates for high-speed, cross-platform frame grabbing).
* **Input Simulation:** `enigo` or `mouce` (Rust crates for simulating native mouse clicks and keystrokes).

## 2. Real-Time Gateway (`/gateway`)
**Core Purpose:** Managing thousands of simultaneous, long-lived bidirectional connections without bottlenecking the AI logic.
* **Core Language:** **Go (Golang)** (Chosen for its lightweight Goroutines and exceptional handling of high-throughput network I/O).
* **Communication Protocol:** **WebSockets** (or gRPC) for maintaining persistent connections with the Rust clients.
* **Message Broker:** **Redis Pub/Sub** (Used to instantly route Wake/Sleep/Execute commands from the Node.js backend to the specific Go instance holding the user's WebSocket connection).

## 3. Core Backend & AI Brain (`/backend`)
**Core Purpose:** Task orchestration, Vision LLM communication, Agentic Loop management, and database operations.
* **Runtime & Language:** **Node.js** with strictly typed **TypeScript**.
* **Framework:** **Express.js** (or NestJS) for exposing REST APIs to the frontend.
* **AI Orchestration:** * **LangGraph.js** or **Vercel AI SDK** (For managing the multi-step ReAct loop and enforcing JSON Structured Outputs).
  * **Vision LLMs:** Supports local/open models (e.g., **Qwen 3.5**) and cloud models (e.g., **Claude 3.5 Sonnet**, GPT-4o).
* **Task Scheduling:** **BullMQ** (Redis-backed message queue for handling delayed, cron, and heavy background tasks reliably).

## 4. Web Dashboard (`/frontend`)
**Core Purpose:** Providing a sleek, responsive, and real-time control center for the human user.
* **Framework:** **Next.js** (App Router).
* **Language:** **TypeScript**.
* **Styling:** **Tailwind CSS** (For rapid, utility-first UI design).
* **State Management:** **React Context** or **Zustand** (For managing live device statuses and streaming the VNC-like desktop view).

## 5. Database & Infrastructure
**Core Purpose:** Persistent storage, long-term memory (RAG), and ephemeral state management.
* **Primary Database:** **MongoDB** (Ideal for storing flexible JSON-like documents, such as complex AI Task Logs and User profiles).
* **Vector Database:** **MongoDB Atlas Vector Search** (Used to store and retrieve embedded user preferences and behavior rules for the RAG persona system).
* **In-Memory Cache & Queues:** **Redis** (Powers BullMQ job queues, session caching, and Go-Node.js Pub/Sub communication).

---

## 🛠️ Summary of Cross-Service Data Formats
* **Images:** Captured in Rust -> Compressed to WebP/JPEG -> Base64 Encoded -> Streamed via Go -> Processed in Node.js.
* **Actions:** JSON payload generated by LLM -> Validated by Node.js -> Routed via Go -> Parsed and executed by Rust.
